# SchedKit Architecture

## Platform Overview

SchedKit is an **API-first incident and assignment coordination platform** вҖ” not a calendar app. The scheduling layer exists to solve a specific ops problem: getting the right person on-scene or on-call as fast as possible, with a full audit trail, SLA tracking, and real-time awareness.

The pitch is intentionally tactical:

> **SchedKit: API-FirstResponder.** Signal in. Assignment out. No clicks required.

Every capability in the platform вҖ” booking, dispatching, incident tracking, org management вҖ” is a first-class API resource. The dashboard is a read/write UI built on the same endpoints your agents and automation scripts call.

### Signals

Signals are the **input layer** вҖ” raw events from devices, operators, or automated systems.

| Type | Description |
|------|-------------|
| `beacon` | Live GPS ping from an active operator (ephemeral вҖ” not persisted per ping) |
| `capture` | Photo or image attached to a field observation |
| `note` | Text note with optional coordinates |
| `checkin` | Manual "I'm here" with location |
| `alert` | High-priority signal вҖ” triggers push notification to org members |

Beacon pings are **not stored per-ping** вҖ” only `beacon_on` (first activation) and `beacon_off` (stop) are persisted for audit. Live position is broadcast via SSE only.

All signals are **org-scoped**: only members of the relevant org receive them on the SSE stream.

### Beacons

Beacons are the continuous-tracking variant of signals. An active beacon:

1. Fires a `beacon` signal on the first ping, writing `beacon_on` to the log
2. Broadcasts live position to org SSE subscribers on every subsequent ping (no DB write)
3. Writes `beacon_off` when stopped, clearing the operator from the live map

The beacon fast-path was a deliberate design decision: GPS pings in field ops can be 1вҖ“5 second intervals. Writing every ping to NocoDB would crush performance.

### Incidents / Tickets

Incidents and tickets are **the same object** in the same database table вҖ” just two mental models over one schema.

- `/v1/tickets` вҖ” for helpdesk/ITSM async flows
- `/v1/incidents` вҖ” for real-time dispatch + SSE war-room flows

Both operate on identical records. Fields:

- `title`, `description`, `status` (`open вҶ’ in_progress вҶ’ resolved вҶ’ closed`)
- `priority` (`low / normal / high / urgent`)
- `source` (`api / email / webhook / alert`)
- `sla_due_at`, `sla_breached`, `sla_status` вҖ” computed SLA window based on priority
- `lat`, `lng`, `location_name` вҖ” incident coordinates
- `customer_token` вҖ” magic link for a public status page (no login required)
- `assignee_id` вҖ” responder assigned to this incident

SLA windows: urgent=1h, high=4h, normal=24h, low=48h.

### Assignments / Bookings

Bookings are the **dispatch output** вҖ” a scheduled assignment linking an attendee (responder) to a host (coordinator) at a specific time.

Booking flow:
1. `GET /v1/slots/:username/:event_slug?date=YYYY-MM-DD` вҖ” available windows
2. `POST /book/:username/:event_slug` вҖ” submit booking
3. Email confirmation sent to attendee; optional host confirmation step
4. `cancel_url` and `reschedule_url` included in all confirmation emails

If `requires_confirmation: true` on the event type, status is held as `pending` until the host clicks confirm/decline from their email.

**Team bookings** (`/v1/book/:org_slug/:team_slug/:event_slug`) auto-assign to an available team member using the team's `routing` strategy: `round_robin` or `random`.

### Orgs / Teams

Multi-tenant scoping works like this:

```
Org (e.g. "Alpha Response")
  в”ңв”Җв”Җ Members (roles: owner / admin / member)
  в””в”Җв”Җ Teams (e.g. "Night Shift")
        в”ңв”Җв”Җ Team Members (active/inactive toggle for rotation)
        в””в”Җв”Җ Team Event Types (bookable via /book/:org_slug/:team_slug/:et_slug)
```

All signals, incidents, and bookings are scoped to an org. SSE streams filter by org membership.

## Technology Stack

| Layer | Technology |
|-------|-----------|
| API server | [Fastify](https://fastify.dev) (Node.js 22) вҖ” ESM, plugin-based |
| Database | [NocoDB](https://nocodb.com) вҖ” REST meta-API over a SQL base |
| Auth | Magic link + 6-digit code вҖ” no passwords, sessions in NocoDB |
| Email | [Mailjet](https://mailjet.com) вҖ” transactional confirmation, invite, and SLA emails |
| Push | Web Push API (VAPID) via `web-push` npm package |
| Real-time | Server-Sent Events (SSE) вҖ” org-scoped, Nginx-buffered off |
| Calendar | Google Calendar API вҖ” optional OAuth per user |
| Payments | Stripe вҖ” coming (plan enforcement hooks already wired) |
| Docs | Mintlify + Swagger/OpenAPI (`@fastify/swagger`) |
| Deploy | Plesk GitOps вҖ” `git push main` вҶ’ webhook вҶ’ `deploy.sh` вҶ’ `pm2 restart` |

## Key Design Principles

### 1. API-First

Every feature is a durable HTTP endpoint before it's a UI element. The dashboard is a consumer of the API, not the other way around. This makes the platform agent-native by default.

### 2. Org-Scoped Everything

No data leaks between orgs. SSE streams, signal feeds, ticket lists, booking access вҖ” all filtered by org membership. There's no global admin query.

### 3. Agent-Native

SchedKit was designed for automation from day one:

- API key auth on every endpoint
- Machine-readable responses (no HTML-only paths except public booking pages)
- Webhook delivery on booking create/cancel
- SSE streams suitable for `EventSource` consumption from any agent runtime
- `customer_token` on incidents for zero-login status polling

### 4. No Click-Ops Deploys

Infrastructure is GitOps only. `git push origin main` triggers Plesk webhook вҶ’ `deploy.sh`. No SSH-and-restart, no Plesk UI deploy buttons.

### 5. Fail Gracefully, Log Everything

- Emails, webhooks, Google Calendar sync, and push notifications are all non-blocking (try/catch, fire-and-forget)
- Core booking/signal operations return success even if downstream delivery fails
- Failures are logged via `fastify.log.error`
